# ガス代相対指数 追加素案

## 1. 目的

各チェーンの「現在のガス代」が、過去 1 週間程度と比べて高いのか安いのかを、ユーザーが一目で判断できる指標を追加する。

現状の UI は「現在値の横比較」には強いが、「そのチェーンの平常時に対して今が高いのか」は判断しづらい。  
そこで、各チェーンごとに直近 7 日間の履歴を基準とした **10 段階の相対指数** を追加する。

---

## 2. 追加したい指標のイメージ

仮称:

- `Chain Stress Index`
- 日本語表示: `チェーン混雑度`

表示例:

- 1: かなり安い
- 5: 平常
- 8: やや高い
- 10: かなり高い

テーブル上の追加列案:

- `Relative`
- `1w Index`
- `混雑度`

表示案:

- 数値: `7/10`
- 補助文言: `やや高い`
- 色: 1-3 緑、4-6 黄、7-8 橙、9-10 赤

---

## 3. 指標の考え方

### 3.1 比較対象

各チェーンについて、現在の `gas_price.gwei` を、過去 7 日間の同一指標の分布と比較する。

基本比較対象:

- 標準モードでは `gas_price.gwei`
- 将来的には `native_fee.wei` ベースにも拡張可能

初期リリースでは、実装と説明の分かりやすさを優先して **`gas_price.gwei` を基準** にするのが良い。

理由:

- 現在の API がすでに返している値と整合する
- `gas_limit` 変更の影響を受けず、チェーン混雑の比較指標として扱いやすい
- LP breaker や ERC-20 の追加指標へ派生しやすい

### 3.2 10 段階化の方法

素直で説明しやすい案は **パーセンタイル方式**。

手順:

1. 各チェーンについて、過去 7 日間の定点観測データをためる
2. 現在のガス代が、その 7 日分の分布のうち何パーセンタイルにあるかを求める
3. その結果を 10 段階に丸める

例:

- 0-10 percentile: 1
- 10-20 percentile: 2
- ...
- 90-100 percentile: 10

この方式の利点:

- チェーンごとの絶対水準差を吸収できる
- 「そのチェーンにとって高いか」が分かる
- ユーザー説明がしやすい

### 3.3 指標ラベル案

| スコア | ラベル案 |
|---|---|
| 1 | かなり安い |
| 2 | 安い |
| 3 | やや安い |
| 4 | 少し安い |
| 5 | 平常よりやや安い |
| 6 | 平常 |
| 7 | やや高い |
| 8 | 高い |
| 9 | かなり高い |
| 10 | 極めて高い |

文言はあとで整理可能だが、最初は以下でも十分:

- 1-3: Low
- 4-6: Normal
- 7-8: High
- 9-10: Very High

## 4. 収集データ案

### 4.1 保存対象

各チェーンごとに以下を時系列で保存する。

- `chain_key`
- `observed_at`
- `gas_price_wei`
- `gas_price_gwei`
- `mode`
- `native_fee_wei`
- `source_note` または `notes`

初期版では最小限でよいので、必須は以下で十分:

- `chain_key`
- `observed_at`
- `gas_price_wei`
- `mode` (`standard` 固定で開始)

`precise` の値は OP/ARB で意味が変わりやすく、履歴基準としてはぶれやすい。  
そのため初期版の基準履歴は **standard モード固定** がよい。

### 4.2 収集頻度

現行の更新周期に合わせて 30-60 秒ごとに取ると粒度は十分だが、7 日分を全部保持すると件数が増える。  
また、**API 呼び出し時に履歴をためる方式は、アクセスが無い時間帯に履歴が欠ける**ため、運用上はやや不安定。

初期案:

- **standard モードの値だけを 5 分おきに 1 サンプル保存**
- 保存は API リクエスト連動ではなく、**バックグラウンド定期ジョブ**で行う

7 日間での件数目安:

- 12 件/時
- 288 件/日
- 2016 件/週/チェーン
- 6 チェーンでも約 12,096 件/週

この規模なら SQLite でも十分運用可能。  
7 日比較が主目的でも、運用では **30-90 日保持して古いデータを日次削除** する形が扱いやすい。

### 4.3 保存先

段階案:

1. 初期版: SQLite
2. 将来: PostgreSQL または Timeseries 系

このサービス規模なら **SQLite を `shared/` 配下に置く案** が最も現実的。

候補:

- `shared/history/gas_history.sqlite3`

メリット:

- Docker でも扱いやすい
- バックアップしやすい
- 追加インフラ不要

補足:

- 単一 API コンテナからのみ書き込む前提なら SQLite で十分
- 複数 API レプリカから同時に書く構成には向かない

### 4.4 運用現実性の評価

結論として、この指標は **スコープを絞れば十分現実的に運用可能**。

現実的な条件:

- 対象は現行 6 チェーンのまま
- 基準は `gas_price.gwei`
- 履歴保存は 5 分間隔
- 保存先は SQLite
- 指標計算は 7 日 percentile

逆に、初回から以下を入れると運用が重くなりやすい:

- `precise` と `standard` の両方を履歴化
- 30 秒ごとの高頻度保存
- chain ごとに複数基準値を同時集計
- API リクエスト依存の履歴収集

この機能の難所は「計算」より **欠損なく履歴をため続けること** なので、運用の成否は保存方式でほぼ決まる。

## 5. API 変更案

既存の `GET /fees/` レスポンス各行に、相対指数情報を追加する。

追加フィールド案:

```json
{
  "relative_index": {
    "score": 8,
    "scale_max": 10,
    "label": "高い",
    "percentile": 0.82,
    "window": "7d",
    "samples": 1842,
    "basis": "gas_price_gwei"
  }
}
```

`FeeRow` への追加案:

- `relative_index.score`
- `relative_index.label`
- `relative_index.percentile`
- `relative_index.window`
- `relative_index.samples`
- `relative_index.basis`

メタ情報追加案:

```json
{
  "meta": {
    "relative_index_enabled": true,
    "relative_index_window": "7d",
    "relative_index_basis": "gas_price_gwei"
  }
}
```

### 5.1 サンプル不足時の扱い

履歴が十分にたまっていない期間は指数を無理に返さない。

例:

- 取得サンプル数が 72 未満なら `null`
- 取得開始から 24 時間未満は `warming_up`
- 返却例:

```json
{
  "relative_index": null
}
```

UI 表示:

- `集計中`
- `—`

返却拡張案:

```json
{
  "relative_index": null,
  "relative_index_status": "warming_up"
}
```

## 7. バックエンド実装案

### 7.1 構成案

追加モジュール候補:

- `api/app/services/history_store.py`
- `api/app/services/relative_index.py`

役割:

- `history_store.py`
  - SQLite への保存
  - 履歴取得
  - 古いデータ削除
- `relative_index.py`
  - 過去 7 日データ取得
  - percentile 計算
  - 10 段階へのマッピング

### 7.2 処理フロー案

`GET /fees/` 実行時:

1. 現在の各チェーンガス代を取得
2. 各チェーンの過去 7 日履歴を読む
3. percentile を計算
4. `relative_index` をレスポンスに付与

バックグラウンドジョブ:

1. 5 分ごとに各チェーンの standard モード値を取得
2. 正常値だけ SQLite に保存
3. 保持期限を超えた古い行を削除

### 7.3 保存タイミング

候補は 2 つ:

1. API リクエスト時に間引いて保存
2. バックグラウンドジョブで保存

**運用を考えると推奨は 2**。

理由:

- アクセス有無に依存せず履歴が育つ
- 7 日比較の説明責任を持ちやすい
- LAN ダッシュボードを閉じてもデータが欠損しない

実装案:

- FastAPI startup で `asyncio.create_task(...)`
- 5 分 sleep で巡回
- shutdown で安全に停止

より分離したい場合の次案:

- sampler 用の別 service / cron コンテナ

ただし現段階では、**API コンテナ内の軽量バックグラウンドタスク**で十分。

## 8. スコア計算の詳細案

### 8.1 第一候補: パーセンタイル法

疑似ロジック:

```text
percentile = 過去7日サンプルのうち current 以下の割合
score = ceil(percentile * 10)
score は 1..10 に丸める
```

補正:

- percentile = 0 でも score は 1
- percentile = 1.0 は score = 10

利点:

- 外れ値に比較的強い
- 実装が簡単

実運用向け補足:

- 保存サンプルは 5 分間隔に固定する
- 初期版の `current` は最新値をそのまま使う
- 必要なら次段階で「直近 3 回中央値」に拡張する

### 8.2 第二候補: 中央値比率法

```text
ratio = current / median_7d
```

たとえば:

- 0.5 未満 = 1
- 0.5-0.7 = 2
- ...
- 2.0 以上 = 10

この案は直感的だが、チェーンごとのばらつき幅を吸収しにくい。  
初期版では **不採用寄り**。

### 8.3 推奨

初期版は **パーセンタイル法を採用**。

必要なら内部で追加保持:

- `median_7d`
- `p90_7d`
- `min_7d`
- `max_7d`

これらを取っておくと、将来の UI 改善に使いやすい。

## 9. 運用上の注意

### 9.1 指標の意味の誤解を防ぐ

この指数は「チェーン間の絶対比較」ではなく、**そのチェーン自身の過去 7 日との比較**であることを UI 上で明示する必要がある。

例:

- ETH が `3/10`
- OP が `8/10`

であっても、絶対的な手数料は ETH の方が高い場合がある。

### 9.2 異常値対策

以下は除外または保守的に扱う:

- RPC 異常値
- 取得失敗後のフォールバック値
- 明らかなゼロ値

方針案:

- `gas_price_wei <= 0` は保存しない
- エラー行は履歴へ保存しない
- `stale=true` の値は保存しない
- 直前保存から 5 分未満の重複保存はしない

追加であるとよい防御:

- chain ごとの異常ジャンプ検知は初期版では不要
- まずは `0`、`None`、エラー、stale の除外だけで十分

### 9.3 チェーン停止・メンテ時

履歴窓の中で欠損が増えても、指数自体は計算可能。ただしサンプル数が少ない場合は信頼度が落ちるので `samples` を返して UI 側で補足する。

## 11. まず決めるべきこと

1. 指標の基準値を `gas_price.gwei` にするか `native_fee` にするか
2. 10 段階化をパーセンタイル法で進めるか
3. 履歴保存を SQLite で始めるか
4. 保存タイミングをバックグラウンドジョブにするか

現時点の推奨:

1. 基準は `gas_price.gwei`
2. 10 段階化はパーセンタイル法
3. 保存先は SQLite
4. 保存タイミングは API コンテナ内の 5 分ごとバックグラウンドジョブ

## 12. コメント

この追加は、単なる「今の金額表示」から一歩進んで、「今は送るべきか」を判断する補助になる。  
特に ETH / Polygon / OP のように日内変動が大きいチェーンでは価値が高い。

一方で、初回から複雑にしすぎると説明負荷が増えるため、まずは:

- 7 日比較
- ガス単価基準
- 10 段階
- パーセンタイル法

の 4 点に絞るのがよい。
